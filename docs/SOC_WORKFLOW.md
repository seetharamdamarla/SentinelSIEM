# SOC Investigation Workflow

This playbook describes the structured workflow a Tier 1 or Tier 2 Security Operations Center (SOC) Analyst should follow when an alert is generated by SentinelSIEM. 

The goal is to ensure alarms are properly assessed, investigated, confirmed as true positive/false positive, and escalated when necessary.

---

## Phase 1: Alert Triage
**Goal:** Determine if the generated alert warrants a full investigation.

1. **Monitor the Wazuh Dashboard:**
   - Log into the dashboard and review the `Security Events` module.
   - Sort by `Level` (Severity 10 and above take priority).
2. **Review the Alert Metadata:**
   - **Timestamp:** When did it occur?
   - **Rule ID:** What rule triggered? (e.g., `100002` Unauthorized File Modification).
   - **Agent Name:** Identify the endpoint (e.g., `macOS-Endpoint-01`).
   - **MITRE ATT&CK Mapping:** Understand the *Tactic* (What is the attacker's goal?) and *Technique* (How are they achieving it?).

## Phase 2: Log Analysis
**Goal:** Gather surrounding telemetry to understand context.

1. **Extract Indicators of Compromise (IOCs):**
   - Source/Destination IP Addresses.
   - Process Names and Arguments.
   - File Hashes (MD5, SHA256 from FIM).
2. **Analyze the Raw JSON:**
   - Click into the specific event in OpenSearch/Discover to read the full log document.
   - *Example:* For a failed SSH attempt, check the `srcip` field. Is it internal or external? Is it a known VPN IP or an unknown subnet?

## Phase 3: Timeline Investigation
**Goal:** Understand the sequence of events before and after the alert.

1. **Broaden the Search:**
   - Search the affected `agent.id` ± 15 minutes around the time of the alert.
2. **Look for the "Kill Chain":**
   - Did a **Brute Force (Rule 100001)** happen first?
   - Followed by a **Successful Login**?
   - Followed by a **Privilege Escalation attempt (Rule 100004)**?
   - Followed by a **FIM Alert (Rule 100002)**?
   - If this sequence exists, it is an active compromise.

## Phase 4: Incident Confirmation (TP vs FP)
**Goal:** Conclusively label the alert.

- **True Positive (TP):** Threat matches malicious intent. Example: The FIM alert was triggered by an unknown process modifying `/etc/shadow`.
- **False Positive (FP):** Authorized or benign activity. Example: The FIM alert was triggered because an IT admin legitimately patched the server, modifying `/etc/hosts`.
  - *Action:* If FP, document the exception and consider tuning the Wazuh rule to ignore the specific admin activity in the future.

## Phase 5: Incident Escalation & Response
**Goal:** Contain the threat and hand off to Incident Response.

1. **Escalate to Tier 3 / Incident Response (IR):**
   - Provide the timeline, extracted IOCs, and a summary of findings.
2. **Containment Actions (if authorized):**
   - Use Wazuh's Active Response capability to block the malicious source IP via firewall (`firewalld-drop` or route null).
   - Isolate the macOS agent from the network.
   - Disable the compromised user account.
3. **Documentation:**
   - Record findings in a ticketing system (Jira, TheHive, etc.).

---

### Example Workflow Scenario: The "Brute Force to FIM" Chain
**1. 09:15 AM:** Analyst sees Level 10 Brute Force Alert (T1110).
**2. 09:20 AM:** Analyst queries the IP `192.168.1.100` and sees thousands of SSH failures.
**3. 09:25 AM:** Analyst expands the timeline and sees a single `Level 3` successful login from `192.168.1.100` for the user `admin`.
**4. 09:26 AM:** Analyst observes a Level 12 FIM Alert for `/etc/passwd`.
**5. Result:** Analyst confirms a True Positive intrusion. Escalates immediately to IR for containment, classifying the severity as CRITICAL.
